# 架构文档生成指南

## 第一部分：使用场景

以下两种场景适合使用 AI 辅助生成或更新架构文档：

### 1. 从零创建架构文档

项目当前没有任何架构文档，需要基于现有代码库从零开始补充：

- 新接手的项目缺少文档传承
- 快速迭代期未及时记录架构设计
- 需要为团队建立统一的架构认知基线

此时应使用**第二部分**的提示词，让 AI 分析代码库并生成完整的架构文档。

### 2. 更新现有架构文档

项目已有架构文档，但存在以下问题：

- 文档内容过于笼统，缺少关键细节
- 随着项目持续开发，文档已落后于实际代码
- 新增模块或重构后文档未同步更新

此时应使用**第三部分**的提示词，让 AI 对比现有文档与当前代码，生成更新后的版本。

---

## 第二部分：从零生成架构文档（无现有文档）

当只有后端代码库，需要从零开始生成文档时，使用以下提示词：

```
你是一位资深软件架构师。请分析以下后端代码库，生成完整的架构文档。

## 任务要求

1. **项目概述**
   - 识别项目类型（API、微服务、单体应用等）
   - 确定主要业务领域
   - 列出技术栈（框架、数据库、外部服务）

2. **目录结构分析**
   - 映射文件夹层级并解释每个目录的用途
   - 识别入口点和配置文件
   - 记录任何非标准的组织模式

3. **模块与组件架构**
   - 识别核心模块及其职责
   - 映射模块间的依赖关系
   - 识别共享工具和通用组件

4. **数据层**
   - 识别数据库类型和使用的 ORM/ODM
   - 列出主要实体/模型及其关系
   - 记录迁移模式（如果存在）

5. **API 层**
   - 按资源/功能分组列出所有 API 端点
   - 识别认证和授权机制
   - 记录中间件栈

6. **外部集成**
   - 列出调用的外部 API 和服务
   - 记录消息队列、缓存系统、文件存储
   - 注明使用的第三方 SDK

7. **配置与环境**
   - 记录环境变量及其用途
   - 识别配置文件及其结构
   - 注明密钥管理方案

## 输出格式

生成一个 Markdown 文档，结构如下：

# [项目名称] 架构文档

## 1. 概述
## 2. 技术栈
## 3. 目录结构
## 4. 核心模块
## 5. 数据架构
## 6. API 参考
## 7. 外部依赖
## 8. 配置说明
## 9. 部署考量
## 10. 架构图（用文本/ASCII 描述）

要详尽、准确，专注于从代码中得出的事实，避免推测。
```

## 第三部分：更新现有架构文档

当已有架构文档需要更新时，使用以下提示词：

```
你是一位资深软件架构师。你的任务是分析后端代码库，并更新现有架构文档以反映当前状态。

## 现有文档

[在此粘贴现有文档]

## 任务要求

将现有文档与当前代码库进行对比，生成更新后的版本：

### 1. 识别变更
- **新增组件**：自上次文档以来新增的模块、服务或功能
- **移除组件**：已废弃或删除的功能
- **修改组件**：现有模块的重大变更
- **配置变更**：新增环境变量、更新的配置项

### 2. 更新各章节
对于现有文档的每个章节：
- 将内容标记为 ✅（准确）、⚠️（需要更新）或 ❌（已过时）
- 用当前事实更新过时信息
- 在缺失处添加新信息

### 3. 保持一致性
- 保留现有文档的结构和格式
- 适当保留有效的历史背景
- 全文使用一致的术语

### 4. 标记破坏性变更
如果有变更会影响使用者或部署：
- 明确标记为 **破坏性变更**
- 说明影响范围和迁移路径（如适用）

## 输出格式

# [项目名称] 架构文档
> 最后更新：[日期]
> 自上次版本的变更：[摘要]

## 变更日志摘要
| 章节 | 状态 | 变更内容 |
|------|------|----------|
| ... | ✅/⚠️/❌ | ... |

[更新后的完整文档内容]

要准确。只记录能从代码中验证的内容。不确定的区域标记为 [需要验证]。
```

---

## 最佳实践

1. **提供足够的上下文**：使用这些提示词时，包含相关的代码文件或文件路径
2. **迭代优化**：先进行广泛分析，再深入具体领域
3. **验证输出**：始终对照实际代码审查 AI 生成的文档
4. **持续维护**：将定期文档更新纳入开发周期
